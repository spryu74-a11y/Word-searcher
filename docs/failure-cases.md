# 검색 안정성 실패 케이스

| ID | 입력값/상황 | 기대결과 | 수정 전 실제결과 |
| --- | --- | --- | --- |
| F-01 | 빈 문자열 또는 공백만 입력 | 검색 요청을 보내지 않고 시작 안내를 표시한다. 버튼은 다시 사용할 수 있다. | 공백을 제거한 뒤 시작 화면으로 돌아가지만, 이전 검색 상태가 남아 있으면 로딩/검색중 표시가 꼬일 수 있다. |
| F-02 | `!!!`, `12345`, `ㄱㄴ` 같은 특수문자/자모/숫자 입력 | 사용자에게 "검색할 한글 또는 영문 단어를 입력해 주세요."를 표시하고 워커 요청을 보내지 않는다. | 문자열 길이만 보고 검색 요청을 보낸 뒤 빈 결과처럼 보인다. 사용자는 입력 오류와 결과 없음 상태를 구분할 수 없다. |
| F-03 | 매우 긴 입력(예: 81자 이상) | 요청을 차단하고 "검색어는 80자 이하로 입력해 주세요."를 표시한다. | 긴 검색어가 그대로 워커와 정규화 로직으로 전달된다. 불필요한 작업과 UI 지연 가능성이 있다. |
| F-04 | 워커/API가 `payload: null`, `results` 누락, `queryInfo` 누락처럼 예상과 다른 구조를 반환 | 기본값으로 보정하고 결과 없음 또는 복구 가능한 오류 메시지를 표시한다. 개발자 콘솔에는 원인 추적용 경고만 남긴다. | `renderSearch()`가 `payload.queryInfo`, `payload.results.length`를 바로 참조해 런타임 예외로 앱이 멈출 수 있다. |
| F-05 | 검색 워커가 오류 또는 타임아웃을 반환 | 진행 중 상태를 해제하고 한국어 오류 메시지와 재시도 버튼을 표시한다. 사용자는 같은 검색을 다시 시도할 수 있다. | 오류 문구만 표시되고 재시도 경로가 없다. 일부 경로에서는 로딩/버튼 상태가 복구되지 않을 수 있다. |
| F-06 | NFD 한글 입력(`가나`) 또는 끝 글자 두음법칙 경계(`계란` -> `란/난`) | NFC로 정규화하여 마지막 글자와 허용 시작 글자를 일관되게 계산한다. | 순수 로직은 일부 처리하지만 UI 입력 검증/문서 계약이 없어 경계 조건 회귀를 잡기 어렵다. |
| F-07 | 빠르게 연속 검색(`가` 입력 후 즉시 `값`) | 이전 요청은 무효화하고 마지막 검색 결과만 렌더링한다. | 취소 로직은 있으나 payload 보정과 상태 초기화가 부족해 늦은 응답/깨진 응답에서 화면이 멈출 수 있다. |
